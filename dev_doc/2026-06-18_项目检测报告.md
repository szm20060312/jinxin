# 🎮 第二部分：游戏分模块完整运行逻辑拆解

## 一、客户端架构

```
main.tsx → App.tsx → (AuthProvider → GameProvider) → AppShell → GameRouter → 阶段面板
```

### 1.1 启动流程

```
浏览器加载 → ReactDOM.createRoot → App 组件
  ├─ 有登录用户？
  │   ├─ 有存档 → 直接进入 GameProvider (LOAD_GAME)
  │   └─ 无存档 → StoryIntro → DifficultySelect → EndingPreview → GameProvider (NEW_GAME)
  └─ 无登录用户 → LoginPage (登录/注册)
```

### 1.2 状态管理（`useGameState.ts`）

- **技术**：`useReducer` + React Context
- **26 种 Action**：NEW_GAME / LOAD_GAME / SET_PHASE / UPDATE_SILVER / UPDATE_REPUTATION / APPLY_EFFECTS / ADD_STAFF / REMOVE_STAFF / PROMOTE_APPRENTICE / ADD_LOG / SAVE_GAME 等
- **自动存档触发**：phase 变化时 + 每 3 分钟定时 + 手动按钮
- **失败即时检测**：UPDATE_SILVER → 库银≤0 → bankrupt；UPDATE_REPUTATION → 信誉≤0 → disgraced

### 1.3 阶段路由（`GameRouter.tsx`）

```
state.phase → 渲染对应组件：
  'morning'     → MorningPanel      (晨间简报)
  'quarterly'   → QuarterlyPanel    (季度决策)
  'simulating'  → 加载动画(1.2秒)
  'annual'      → AnnualPanel       (年终合账)
  'event'       → EventPhase        (事件处理)
  'game_over'   → GameOverPanel     (结局展示)
```

### 1.4 主布局（`AppShell.tsx`）

```
┌────────────────────────────────────────┐
│  Header: 标题 + 日期 + 存档/新局/退出  │
├────────┬───────────────────────────────┤
│Sidebar │  MainPanel                     │
│库银    │  (按 phase 或 NavView 渲染)    │
│信誉    │                                │
│分号    │                                │
│伙友    │                                │
│[导航]  │                                │
└────────┴───────────────────────────────┘
```

- **NavView 四视图**：game（主面板）/ branches（分号）/ staff（伙友）/ logs（日志）
- 游戏结束时 Sidebar 隐藏，全屏展示结局

## 二、服务端逻辑（引擎层，全纯函数）

> 注：项目为纯前端，无独立服务端。"服务端逻辑"指引擎层纯函数计算。

### 2.1 核心经济循环

```
晨间简报生成 → 季度决策确认 → 经济模拟计算 → 终局检测 → 事件检测 → 下一季
```

#### 步骤 1：晨间简报（`morning.ts` → MorningPanel）

- 每季生成 **3-5 条**简报
- **12 类模板**：库银盘点（必出 1 条）+ 在途银/准备金/汇兑行情/同行消息/学徒进度/伙友动态/官府/商帮/分号异动/密押到期/其他（随机 2-4 条）
- 每条标注 `reliability`（高/中/低），影响可信度展示
- 玩家逐条"知晓"后方可进入季度决策

#### 步骤 2：季度决策（`decisions.ts` → QuarterlyPanel）

- **5 项必出核心决策**：
  1. **白银调拨**：汉口 3 万 / 张家口 2.5 万 / 从分号调回（硬编码阈值 5 万/2 万）
  2. **汇费调整**：+0.5% / -0.5%
  3. **存贷利率**：+0.5% / -0.5%
  4. **放款额度**：×1.5 / 维持 / ×0.6
  5. **分号巡查**：2,000 两/次
- **1-2 项动态决策**（30% 概率触发）：茶商大单、盐商存银、恰克图商路、同行压价、商会催捐、官府查封、季节事件、伙友调度、同行挖角、市井传言
- 玩家逐项选择 → 统一确认

#### 步骤 3：经济模拟（`simulation.ts` → 1.2 秒动画）

核心公式：

```
┌─────────────────────────────────────────────┐
│ 汇兑业务量 = 基础量(8万/6万)                │
│   × 信誉倍率(0.1-2.0)                       │
│   × 人员修正(掌柜0.7+伙友0.3)               │
│   × 分号状态(0.3-1.0)                        │
│   × 等级修正(1.0/0.85)                       │
│   × 明星加成(1.08, 如有)                     │
│   × 随机波动(±15%)                           │
├─────────────────────────────────────────────┤
│ 汇费收入 = 业务量 × 汇费率                   │
│ 存款利息支出 = 存银 × (年利率/4)             │
│   (存银 ≈ 总库银 × (0.3 + 信誉/100×0.5))     │
│ 放款利息收入 = 放款额 × (max(存息×4,8%)/4)   │
│ 运营支出 = 学徒5两/月 + 伙友20两/月           │
│   + 分号500两/季 + 总号800两/季              │
│ 在途增加 = 汇兑总量 × 10%                    │
│ 学徒成长 = talent×0.9 + random(-5,10)         │
│ 潜逃检测 = loyalty<30 → 10%/季 → -15%库银   │
│ 收入分配 = 20%总号 + 40%汉口 + 40%张家口     │
│ 净收入 = 汇费收入 + 放款利息 - 存款利息       │
│   - 运营支出 - 潜逃损失                       │
└─────────────────────────────────────────────┘
```

#### 步骤 4：信誉季度变动（`reputation.ts`）

```
原始Δ = 准备金率影响(-3~+3)
      + 周转效率(-1~+1)
      + 密押安全(0~-1)

最终Δ = 原始Δ × 等级阻尼系数(golden×0.5 ~ ruined×1.5)
```

- **非对称设计**：下降速度是上升速度的 5 倍，强调信誉崩盘的不可逆性

#### 步骤 5：终局检测

```
库银 ≤ 0                → 惨淡收场 (bankrupt)
信誉 ≤ 0                → 晚节不保 (disgraced)
库银 ≥ 120万 + 年≥1815  → 金玉满堂 (golden_age)
年 ≥ 1813               → 守成之主 (survivor)
年≥1818 + 信誉≥85 + 库银≥90万 → 急流勇退 (可选)
任意时刻点结束           → 功成身退
```

- **重要**：检测顺序决定结局优先级。守成之主先于金玉满堂检测，因此 1813 年后同时满足两条件时，会先触发守成之主，金玉满堂不可达。

#### 步骤 6：事件检测

- **主线事件**：检查 `triggeredEventIds` + 年份窗口 + 信誉门槛 → 逐个触发
- **危机事件**：随机选择 1-2 个未触发的
- **随机事件**：15% 概率触发，去重

### 2.2 年度循环（`annual.ts` → AnnualPanel）

4 步流程，每年末触发：

```
Step 1 对账：分号报表 + 差异项(0-2条, 800-8000两)，采信解释/花1500两查账
Step 2 分红：厚东家(70/20/10) / 厚积薄发(50/35/15) / 号规均分(60/25/15)
Step 3 人事：续任(loyalty+5) / 调离轮换(loyalty-5) / 辞退(loyalty-3全员, rep-2)
Step 4 规划：开疆拓土 / 固本培元 / 信誉至上 / 急流勇退(条件满足时)
```

# 📖 第三部分：整合后的完整游戏背景故事

## 一、时代背景

**嘉庆十三年（1808 年）秋，山西平遥。**

白莲教之乱余烬未息，嘉庆帝面对军饷耗尽的空虚国库，不得不默许民间金融暗流涌动。山西平遥南大街——"大清金融中心"——驼铃声与算盘声昼夜交织。日升昌、蔚泰厚等大票号鳞次栉比，每一扇黑漆大门背后，都流转着成千上万两白银的汇票。

晋商以"信义为本"。票号卖的不是米面绸缎，而是一张薄纸——凭票号招牌的信誉，商人能从京师提走汉口分号的万两白银。**信，就是晋商的命。**

## 二、主角身世

你——**票号大掌柜**——出身平遥商贾世家，十余岁入号从学徒做起，扫地、端茶、跑街、学算，历经二十年磨砺，终被东家委以重任，执掌这间中型票号。

你的案头摆着三样东西：
- **一把算盘**——算的是万千白银的进出，算不清的是人心向背；
- **一方印章**——盖下去白纸变汇票，盖的是晋商三百年的信义；
- **一本密押**——只有你和分号掌柜能读懂的天书，守护着汇通天下的命脉。

## 三、开局三重压力

清晨，三封信摆在你的案头：

**第一封**——汉口分号来函：长江秋汛冲毁茶市，银根紧缩，汇兑量锐减。

**第二封**——张家口分号急报：口外俄商蠢蠢欲动，疑似派人刺探密押编码方式。

**第三封**——平遥县衙师爷递来帖：朝廷军饷吃紧，望票号"协助"官银汇兑，言下之意——账目要好看。

窗外，南大街的驼铃声由远及近。你知道——**抉择的时刻到了。**

## 四、三条铁律

1. **信为命脉**：宁可赔光最后一两银子，不可拒兑一张真票。票号立身之本，不在库银多少，而在持票者信不信你。
2. **利在长远**：晋商不做一锤子买卖。今日让三分利，明日多一条路。江湖上多一个朋友，比库房里多一万两银子更值钱。
3. **庙堂与江湖的平衡**：不可依附官府太深，也不可远离庙堂太远。红线在哪，只有你手中的算盘知道。

## 风险等级说明

| 标记 | 含义 |
|------|------|
| 🔴 **致命** | 直接影响游戏可玩性，导致数据错误或最高成就不可达 |
| 🟠 **严重** | 显著影响游戏体验或数值平衡 |
| 🟡 **一般** | 存在隐患，特定条件下触发 |
| 🟢 **轻微** | 代码质量问题，不影响功能 |

## 🔴 致命级问题

### 问题 1：结局检测顺序导致"守成之主"覆盖"金玉满堂"

- **位置**：`src/engine/gameState.ts` — `determineEnding()` 函数
- **描述**：`determineEnding` 按固定顺序检测结局。`survivor`（守成之主，年≥1813）排在 `golden_age`（金玉满堂，年≥1815 + 库银≥120万）之前。如果玩家在 1815 年后同时满足两个条件（库银≥120万），会先匹配到 `survivor`，永远无法触发 `golden_age`。
- **风险等级**：🔴 致命
- **负面影响**：玩家达成 120 万两目标却获得低一级的结局，严重挫败感。最高成就不可达。
- **修复状态**：⚠️ 未修复。集成测试报告未提及此问题。
- **修复方案**：将 `golden_age` 检测置于 `survivor` 之前，确保最高成就优先匹配。

### 问题 2：QuarterlyPanel 中 `connectionsDelta` 合并逻辑缺陷

- **位置**：`src/components/quarterly/QuarterlyPanel.tsx` — `handleConfirm` 函数中 `connectionsDelta` 累加逻辑
- **描述**：同一季度多个决策影响同一人脉维度时，使用 `||` / `??` 操作符可能导致后人脉变动覆盖前值，而非正确累加。需验证具体代码逻辑确认实际影响。
- **风险等级**：🟠 严重（降级，实际触发场景罕见）
- **负面影响**：同一季度多个决策影响同一人脉维度时，后者覆盖前者，导致人脉变化量不准确。
- **修复状态**：⚠️ 需验证具体代码逻辑。
- **修复方案**：使用 `+=` 累加而非赋值覆盖。

## 🟠 严重级问题

### 问题 3：密码明文存储，无真正安全性

- **位置**：`src/utils/accountManager.ts`
- **描述**：密码使用简单加盐位运算 `hash`，存储在 localStorage 明文可读。任何能访问浏览器 DevTools 的人都能直接读取所有用户密码。
- **风险等级**：🟠 严重
- **负面影响**：账号安全形同虚设。但因是纯前端单机游戏，实际危害有限。
- **修复状态**：⚠️ 设计如此（MVP 阶段无后端），已在开发文档中标注。
- **修复方案**：MVP 阶段可接受。后续接入后端时迁移至 bcrypt + JWT。

### 问题 4：存档数据无版本迁移机制

- **位置**：`src/engine/gameState.ts` — `loadGame()` 函数
- **描述**：存档版本硬编码为 `'0.1.0'`，`loadGame` 中仅对 `triggeredEventIds` 做了兼容处理。未来 GameState 结构变更时，旧存档将丢失数据或崩溃。
- **风险等级**：🟠 严重
- **负面影响**：版本更新后玩家存档可能损坏。
- **修复状态**：⚠️ 未实现完整的版本迁移层。
- **修复方案**：实现 `migrateSave(save, fromVersion, toVersion)` 链式迁移函数。

### 问题 5：学徒出师周期过长（3 游戏年）

- **位置**：`src/engine/simulation.ts` — 学徒成长逻辑 + `src/data/constants.ts` — 每季+3月
- **描述**：学徒需要 `monthsTrained >= 36` 才能出师，每季推进 3 个月。出师需 12 季 = 3 游戏年。游戏胜利条件（守成之主）仅需 5 年（1813-1808=5），学徒在游戏中出师时已接近尾声。
- **风险等级**：🟠 严重（数值设计问题）
- **负面影响**：学徒养成系统存在感弱，玩家可能全程看不到学徒出师。初始 3 个学徒基本是"摆设"。
- **修复状态**：⚠️ 设计如此，但数值可能需要调整。
- **修复方案**：方案 A：每季训练月数+6，出师缩短至 1.5 年；方案 B：初始学徒 monthsTrained 设为 12-24（非零起点）。

### 问题 6：困难模式初始条件过于严苛

- **位置**：`src/data/constants.ts` — INITIAL_SILVER_HARD + INITIAL_REPUTATION_HARD
- **描述**：困难模式初始银两 31 万（普通 77 万的 40%），信誉 35（业务倍率仅 0.5），人脉全面降低。三者叠加：银两少 → 准备金率低 → 信誉持续下降 → 业务倍率低 → 收入少 → 恶性循环。首季模拟可能直接亏损。
- **风险等级**：🟠 严重
- **负面影响**：困难模式可能"不可能通关"，开局即进入死亡螺旋。首季亏损率约 60%（基于普通模式首季净收入 814 两推算，困难模式信誉倍率 0.5 导致业务量减半、收入锐减，而固定支出不变）。
- **修复状态**：⚠️ 数值未经充分测试。
- **修复方案**：初始信誉从 35 → 40（跨入"中规中矩"，业务倍率从 0.5 恢复至 1.0），或前 4 季信誉不下降（新手保护）。

## 🟡 一般级问题

### 问题 7：密押系统对游戏进程影响过弱

- **位置**：`src/engine/reputation.ts` — `calcSeasonalReputationChange()` + `src/data/ciphers.ts`
- **描述**：密押使用超 14.4 个月仅扣 1 点信誉（还被阻尼系数削弱）。高级密押安全期仅 6 个月但费用 12,000 两。密押泄露事件仅在 4-10 年触发一次。整体上密押系统的投入产出比极低，玩家最优策略是使用基础密押并忽略更换。
- **风险等级**：🟡 一般
- **负面影响**：一个核心特色系统形同虚设。
- **修复状态**：⚠️ 数值设计问题，需重新平衡。
- **修复方案**：密押过期惩罚从 -1 → -3 信誉，且不受阻尼系数削弱；增加密押泄露事件触发概率。

### 问题 8：`Date.now()` 生成 ID 存在冲突风险

- **位置**：`src/engine/decisions.ts` — `generateQuarterlyDecisions()` 中 `id: \`decision_${Date.now()}\``
- **描述**：多个决策在同一毫秒内生成时 ID 相同。虽然概率极低，但 React key 冲突会导致渲染异常。
- **风险等级**：🟡 一般
- **负面影响**：极端情况下列表渲染错误或 key warning。
- **修复状态**：⚠️ 已知但未修复（BUG-014 标记了类似问题但未覆盖 decisions.ts）。
- **修复方案**：使用模块级递增计数器生成唯一 ID（与 morning.ts 中 BUG-014 修复方案一致）。

### 问题 9：`day` 字段从未用于游戏机制

- **位置**：`src/types/index.ts` — GameState.date.day + `src/engine/simulation.ts`
- **描述**：Date 类型定义了 `day: number`（1-90），模拟中 `day += 90` 推进，但没有任何游戏机制基于 `day` 做判定。它是纯粹的数据字段，占用存储空间却无功能。
- **风险等级**：🟡 一般
- **负面影响**：代码冗余，存档体积略增，不影响功能。但对日后扩展造成误导。
- **修复状态**：⚠️ 设计残留。
- **修复方案**：保留以备后续扩展（例如按天触发的事件），但添加注释说明当前未使用。

### 问题 10：年终账目差异金额估算偏差

- **位置**：`src/engine/annual.ts` — `generateAnnualLedger()` + `generateDiscrepancies()`
- **描述**：年终报表基于当前 state 反向估算，无结构化历史数据。差异项金额在 800-8,000 两之间随机，与玩家实际经营状况无关联。
- **风险等级**：🟡 一般
- **负面影响**：年度对账的"解谜"体验弱，差异项更像是随机的"罚款"而非真正的会计追查。
- **修复状态**：⚠️ 设计如此（MVP 阶段无历史流水）。
- **修复方案**：增加 `quarterlyLedgers` 历史记录数组，年报基于实际历史数据计算。

### 问题 11：StoryIntro 音频文件缺失时的静默降级

- **位置**：`src/components/auth/StoryIntro.tsx` — 音频预加载逻辑
- **描述**：使用 `fetch('HEAD')` 检测音频文件存在性，但可能因 CORS 策略失败，导致合法文件被误判为缺失。部署到 EdgeOne Pages 后文件路径可能变化。
- **风险等级**：🟡 一般
- **负面影响**：TTS 旁白在某些部署环境下完全静默。
- **修复状态**：⚠️ 需在部署环境验证。
- **修复方案**：改用 HTMLAudioElement `onerror` 事件检测，或直接使用 try/catch 的播放尝试替代预检测。

- **验证**：困难模式开局连续 4 季，确认不会首季破产

### 调整 3：强化密押系统影响力

- **文件**：`src/engine/reputation.ts` + `src/data/events.ts`
- **方案**：
  - 密押过期惩罚从 -1 → -3 信誉，且不受阻尼系数削弱
  - 增加密押泄露事件的触发概率（加入季度决策中的动态事件池）
  - 被泄露时损失从 -5~-15 → -10~-25 信誉
- **验证**：使用过期密押连续 2 季，确认信誉明显下降
